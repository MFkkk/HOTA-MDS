# 技术决策 / DECISIONS

本文档记录开发过程中“为什么这么做”，用于补充 `SPEC.md`、`PLAN.md`、`STATUS.md`、`HANDOFF.md` 之外的实现决策背景。

---

## 2026-04-13

### 决策：一期范围先只做外部参观双屏大屏，不把内部 Web 报表纳入一期
原因：
- PRD 和 SPEC 都明确一期聚焦外部参观场景
- 内部报表需求、权限和统计口径尚未稳定
- 先收敛范围，避免一期目标被持续扩张

影响：
- 一期开发不进入内部报表页面和报表 API
- 二期需要单独再做 PRD、模型和权限规划

### 决策：报修与 3D 仿真不作为一期前段阻塞项
原因：
- 报修系统取数条件不稳定
- 3D 仿真优先级低于一期核心展示链路
- 一期验收关键仍是设备、产量、排产、能耗稳定展示

影响：
- 一期前段只保留报修占位和 3D 预留区
- 报修和 3D 的真实接入放到后续阶段

### 决策：前端不直连外部系统，统一由后端拉取、标准化、缓存后提供 API
原因：
- 可屏蔽 SAP、排产库、能耗库、OPCUA 等外部系统差异
- 有利于统一鉴权、错误处理和缓存策略
- 满足“数据源异常时大屏显示最近一次成功数据”的要求

影响：
- 前端只依赖本系统 API，不依赖外部系统结构
- 后端需要承担定时拉取、标准化、缓存和健康状态管理

## 2026-04-14

### 决策：后台权限一期先只做管理员权限，不做复杂 RBAC
原因：
- 当前阶段目标是尽快具备最小后台维护能力
- 一期只有后台管理场景，复杂角色拆分收益有限
- 可以降低模型、接口和前端管理复杂度

影响：
- 当前后台接口统一要求管理员身份
- 后续如果进入多角色体系，需要在现有接口上补权限层

### 决策：管理员认证当前采用 Django 签名 Bearer Token，而不是直接引入 JWT
原因：
- 当前是 M2 最小后台能力，优先保证实现简单、可用、依赖少
- 可直接复用 Django 用户体系和签名能力
- 当前还没有网关、刷新令牌、多端会话等复杂诉求

影响：
- 前端仍要处理登录态过期和重新登录
- 如果后续接入统一网关或更复杂认证体系，可能迁移到 JWT 或集中认证方案

### 决策：后台接口统一使用 `success/code/message/data` 响应包裹格式
原因：
- 便于前后端统一处理成功和失败场景
- 能减少不同模块返回风格不一致的问题
- 有利于后续继续扩展管理接口和展示接口

影响：
- 新增 API 时应保持同样结构
- 需要统一异常处理，避免直接漏出框架默认错误格式

### 决策：数据源敏感配置采用“结构预留”方式，而不是当前阶段直接落地真实加密方案
原因：
- 一期当前还未接真实外部系统，真实密钥流转方案尚未最终确认
- 但数据模型必须提前体现环境变量引用和密文存储两类能力
- 可以先保证接口与模型不会把敏感信息写死在代码里

影响：
- 当前 `DataSourceConfig` 支持 `env_ref` 和 `encrypted` 两种结构
- 真正的密文加解密机制、密钥轮换和运维流程后续仍需补齐

### 决策：M2 先提供最基础的后台前端界面，而不是继续只堆 API
原因：
- 只有 API 而没有最小界面，后台能力无法被直接使用和验证
- 最小界面可以更早暴露字段设计和接口设计问题
- 有助于后续 M3/M4 开发时直接维护配置和 mock 数据

影响：
- 当前已提供 `/admin/login` 和 `/admin/console`
- 该界面以可用为主，不做复杂交互和视觉包装

### 决策：根据数据库结构审阅结果补充员工表，并将员工角色先收敛为固定枚举
原因：
- 当前数据库结构缺少员工台账，不利于后续后台维护和业务扩展
- 本轮需求已经明确员工号格式和角色范围，无需等待更多外部系统资料
- 一期仍应保持简单，不引入复杂组织架构、岗位体系或权限继承

影响：
- 后台新增员工台账模型、API 和前端管理入口
- 员工号当前限制为英文和数字，禁止特殊字符
- 角色当前固定为 `employee`、`team_leader`、`supervisor`，后续如扩展需评估兼容性
## 2026-04-14（员工台账补充收尾）

### 决策：员工号当前仅允许英文和数字
原因：
- 这是本轮用户明确提出的输入约束。
- 先收紧格式，可以减少后续与外部编码、条码或工号体系对齐时的歧义。
- 当前阶段只做最小后台能力，不引入更复杂的编号兼容逻辑。

影响：
- 后端对 `employee_no` 增加正则校验，禁止空格、连字符和其他特殊字符。
- 前端员工台账录入需要遵循同样约束。
- 如果后续现场存在带分隔符的工号，需要评估兼容策略并安排数据迁移。

### 决策：员工角色先收敛为固定枚举，不做可配置角色体系
原因：
- 当前一期后台只做管理员权限，不做复杂 RBAC。
- 本轮只需满足员工、班组长、主管三类角色展示和维护。
- 先固定枚举可以保持 API、前端表单和数据库结构简单一致。

影响：
- 当前角色值固定为 `employee`、`team_leader`、`supervisor`。
- 前后端均按固定选项实现，不开放后台自定义角色。
- 若后续出现更多岗位类型，需要评估是否升级为独立角色主数据。

### 决策：员工台账先作为独立主数据维护，不与后台登录账号绑定
原因：
- 本轮目标是补足数据库结构中的缺口，而不是扩展认证或人员体系。
- 员工主数据与管理员登录账号并不是同一问题域，强行绑定会扩大范围。
- 这样可以保持本轮只完成一个明确的最小可交付目标。

影响：
- `Employee` 只承载台账信息，不参与当前管理员登录认证。
- 后续如需做员工自助登录、审批链或组织关系，再单独设计关联模型。

## 2026-04-20

### 决策：M3 先用标准化缓存快照 + mock 装载命令跑通展示接口，不提前接真实数据源
原因：
- 当前外部系统点表、字段和连接方式仍未到位，但前后端都需要稳定的标准展示接口。
- PRD、SPEC、PLAN、STATUS、HANDOFF 都要求当前阶段不要让前端直连外部系统，并优先保证“最近一次成功数据兜底”。
- 先把缓存快照、mock 装载和展示 API 跑通，可以在不扩大范围的前提下验证标准模型与异常策略。

影响：
- 当前左右屏接口 `/api/screens/left`、`/api/screens/right` 先读取标准化快照，而不是读取任何外部系统原始结构。
- 当前提供 `load_mock_screen_data` 管理命令，用于装载 mock 快照和模拟失败场景。
- 后续进入 M5 时，真实采集任务应复用这批标准快照模型写入数据，而不是绕过缓存层直接改大屏接口结构。

### 决策：展示配置中的历史脏文案优先在后端读取层做兜底，而不是让前端分散处理
原因：
- 当前已有历史坏值进入 `DisplayContentConfig`，典型表现为公司名或欢迎语只剩 `?`。
- 这类问题属于展示数据质量问题，应该由后端在标准化输出阶段统一兜底，而不是让多个前端页面分别判断。
- 后端统一兜底更符合“后端负责标准化、前端负责展示”的项目规则。

影响：
- 当前左/右屏接口在读取展示内容配置时，会自动把仅由问号组成的坏值回退为默认中文文案。
- 前端页面不需要再为这类历史脏值增加重复判断逻辑。
- 后续若出现更多展示类脏值，应优先继续补在后端标准化层，而不是在具体页面修补。

### 决策：双屏中的状态枚举和占位说明继续下沉到后端标准化输出，前端只消费展示字段
原因：
- 左屏 `running/stopped/alarm/offline`、右屏 `normal/warning/delayed/paused` 这类值本质上属于内部状态键，不适合直接暴露给外部参观者。
- 如果由前端自行维护这些映射，后续多个页面或多个展示端容易出现文案不一致。
- 当前项目规则明确要求后端负责标准化、缓存并向前端提供 API。

影响：
- 左屏接口新增 `deviceOverview.statusItems`，右屏接口新增 `schedule.riskSummary.items`，统一提供可直接展示的 `label`、`count`、`accent` 等字段。
- 前端 `ScreenDisplay` 改为直接渲染这些标准化字段，不再自己解释业务枚举。
- 后续继续扩展双屏字段时，应优先沿用这种“后端给展示语义，前端只负责布局渲染”的模式。

### 决策：报修与 3D 在一期仍保持非阻塞占位，但占位说明也由后端统一提供
原因：
- 报修与 3D 仿真当前仍不是一期前段阻塞项，但展示页需要明确说明其阶段定位，避免参观时产生误解。
- 如果占位说明写死在前端，后续阶段描述调整时需要多处同步修改。
- 将占位说明也纳入接口返回，可以保持左右屏展示语义统一可控。

影响：
- 当前接口新增 `repairPlaceholder.description` 与 `simulationPlaceholder.description`。
- 前端不再硬编码这两块说明文案。
- 后续如果项目阶段定位变化，只需调整后端输出或配置来源，不需要分别修改每个展示页。
